## Use When

Incoming issues, bugs, or feature requests need triaging, prioritising, and routing to the right state.

## Core Concept

Every issue moves through states with clear transition rules:

```
unlabeled ──► needs-triage ──► needs-info ──► ready-for-agent ──► ready-for-human
                   │                                               │
                   └──────────► wontfix ◄──────────────────────────┘
```

| State | Description | Entry Action |
|-------|-------------|--------------|
| **unlabeled** | No state label applied | Apply `needs-triage` label |
| **needs-triage** | Needs evaluation | Attempt reproduction (bugs) or evaluate scope (enhancements) |
| **needs-info** | Waiting for reporter response | Note what info is needed; when reporter replies, return to needs-triage |
| **ready-for-agent** | Fully specified, agent can implement | Write agent brief |
| **ready-for-human** | Needs human judgement | Note delegation blockers |
| **wontfix** | Rejected | Write out-of-scope doc or close with explanation |

## Precise Vocabulary

| Term | Definition |
|------|------------|
| **unlabeled** | Issue without a state label applied |
| **needs-triage** | Issue needing evaluation — first actionable state |
| **needs-info** | Issue waiting for reporter response |
| **ready-for-agent** | Fully specified issue an agent can implement |
| **ready-for-human** | Issue requiring human judgement |
| **wontfix** | Rejected issue |
| **agent brief** | Specification document written when moving to ready-for-agent |
| **out-of-scope doc** | Decision record written when marking an enhancement as wontfix |

## Context Requirements

- Access to issue tracker with label management
- `.out-of-scope/` directory for prior rejection records
- Codebase access for reproduction and scope evaluation

## Workflow

### 1. Show What Needs Attention

Present issues in 3 buckets, oldest first within each:

- **Unlabeled** — new issues without a state label
- **Needs-triage** — ordered by reported severity (bugs above enhancements)
- **Needs-info with activity** — reporter has replied since last triage note

### 2. Triage a Specific Issue

For each issue triaged:

1. **Gather context**: read the full issue text, search the codebase for relevant code paths, check `.out-of-scope/` for any prior rejection on the same concept
2. **Recommend category + target state**: classify as `bug` or `enhancement`, propose the target state with written reasoning
3. **Attempt reproduction (bugs only)**: reproduce the issue → rule out environment factors → narrow to minimal reproduction
4. **Grill if uncertain**: ask clarifying questions one at a time until the issue is understood sufficiently to assign a state
5. **Apply outcome**: move the issue to the decided state

Every AI-generated comment on the issue MUST start with:

> *This was generated by AI during triage.*

### 3. Quick State Override

If a maintainer says "move #42 to ready-for-agent", trust their judgement, confirm the action, and execute directly. Flag unusually permissive transitions (e.g. unlabeled → ready-for-agent) with a brief note.

### State Transition Rules

- **unlabeled → needs-triage**: Required first step. Cannot skip directly to any other state.
- **needs-triage → needs-info**: When more information is needed from the reporter. Be specific about what's missing.
- **needs-info → needs-triage**: When the reporter replies with the requested information.
- **needs-triage → ready-for-agent**: Issue is fully understood, specified, and scoped. Write an agent brief.
- **needs-triage → ready-for-human**: Issue requires human judgement (design decision, product call, security review).
- **needs-triage → wontfix**: Rejected. Be conservative — prefer needs-info over wontfix if uncertain.
- **ready-for-agent → ready-for-human**: Agent encountered a decision it cannot make. Delegate explicitly.
- **ready-for-agent → needs-triage**: Implementation revealed ambiguity. Re-triage.

### Agent Brief Template

Write this when moving an issue to **ready-for-agent**:

```markdown
## Agent Brief: #[issue] [Title]
Parent: #[parent-issue] | Label: ready-for-agent

### What to build
[Behavioral description, not procedural]

### Acceptance criteria
- [ ] Criterion 1
- [ ] Criterion 2

### Scope boundaries
- In scope: [list]
- Out of scope: [list]

### Reference
- Files: [paths — durable, describe interfaces not file paths]
```

### Out-of-Scope Knowledge Base

When an enhancement is marked **wontfix**, archive the decision so future requests can be handled quickly:

1. Create `.out-of-scope/<concept>.md` with:
   - Concept name
   - Decision: wontfix
   - Reason (why it was rejected)
   - Prior requests (issue numbers)

2. Link the out-of-scope doc in a closing comment

3. On subsequent requests for the same feature, reference the existing doc before closing

## Tool Requirements

- **terminal**: issue tracker CLI (e.g. `gh`) for label management
- **read**: read issue text, out-of-scope docs, codebase files
- **grep**: search codebase for relevant code paths
- **glob**: find files by pattern
- **edit**: create and update out-of-scope docs, agent briefs
- **write**: write new documents

## Output

- Triage decisions with state transitions applied
- Agent briefs for issues promoted to ready-for-agent
- Out-of-scope decision records for wontfix enhancements
- Closing comments with explanations for every rejection

## Quality Criteria

- Every AI-generated comment MUST start with: `> *This was generated by AI during triage.*`
- unlabeled → needs-triage first (cannot skip directly to ready-for-agent)
- needs-info returns to needs-triage when reporter replies
- Maintainer can override anytime (flag unusual transitions)
- Be conservative with wontfix — prefer needs-info if uncertain
- Never close without explanation (bugs) or out-of-scope doc (enhancements)

## When NOT to Use

- Issues already in ready-for-agent with a valid agent brief
- Issues requiring immediate human judgement (design decisions, product calls, security reviews) — route directly to ready-for-human
- Issues confirmed outside the project's scope
