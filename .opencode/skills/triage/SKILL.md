# Triage — Route & Prioritize Incoming Issues

**Tradeoff:** Speed vs thoroughness. Fast triage keeps pipeline moving but risks misrouting. Deep analysis catches nuance but stalls critical issues. Default to velocity — escalate when uncertain.

## Core Concept

Every issue moves through states with clear transition rules:

```
unlabeled ──► needs-triage ──► needs-info ──► ready-for-agent ──► ready-for-human
                   │                                               │
                   └──────────► wontfix ◄──────────────────────────┘
```

| State | Description | Entry Action |
|-------|-------------|--------------|
| **unlabeled** | No state label applied | Apply `needs-triage` label |
| **needs-triage** | Needs evaluation | Attempt reproduction (bugs) or evaluate scope (enhancements) |
| **needs-info** | Waiting for reporter response | Note what info is needed; when reporter replies, return to needs-triage |
| **ready-for-agent** | Fully specified, agent can implement | Write agent brief |
| **ready-for-human** | Needs human judgement | Note delegation blockers |
| **wontfix** | Rejected | Write out-of-scope doc or close with explanation |

## Precise Vocabulary

| Term | Definition |
|------|------------|
| **unlabeled** | Issue without a state label applied |
| **needs-triage** | Issue needing evaluation — first actionable state |
| **needs-info** | Issue waiting for reporter response |
| **ready-for-agent** | Fully specified issue an agent can implement |
| **ready-for-human** | Issue requiring human judgement |
| **wontfix** | Rejected issue |
| **agent brief** | Specification written when moving to ready-for-agent |
| **out-of-scope doc** | Decision record written when marking enhancement wontfix |

## Context Requirements

- Issue tracker with label management
- `.out-of-scope/` directory for prior rejection records
- Codebase access for reproduction and scope evaluation

## Workflow

### 1. Show What Needs Attention

Present issues in 3 buckets, oldest first:

- **Unlabeled** — new issues without a state label
- **Needs-triage** — ordered by severity (bugs above enhancements)
- **Needs-info with activity** — reporter replied since last triage note

### 2. Triage a Specific Issue

1. **[Gather context]** Read full issue, search codebase for relevant paths, check `.out-of-scope/` for prior rejections → verify: all context sources checked before routing
2. **[Categorize]** Classify as `bug` or `enhancement`, propose target state with reasoning → verify: category + target state explicitly stated
3. **[Reproduce (bugs)]** Reproduce → rule out env factors → narrow to minimal repro → verify: repro steps documented or clear explanation why not replicable
4. **[Grill if uncertain]** One question at a time until issue understood → verify: question asked, response received, state assigned
5. **[Apply outcome]** Move issue to decided state → verify: label applied, comment posted

Every AI comment on the issue MUST start with:

> *This was generated by AI during triage.*

### 3. Quick State Override

Maintainer says "move #42 to ready-for-agent" → trust, confirm, execute. Flag unusually permissive transitions (e.g. unlabeled → ready-for-agent) with brief note → verify: override documented.

### State Transition Rules

- **unlabeled → needs-triage**: Required first step. Cannot skip directly to any other state.
- **needs-triage → needs-info**: More info needed from reporter. Be specific.
- **needs-info → needs-triage**: Reporter replied with requested info.
- **needs-triage → ready-for-agent**: Fully understood, specified, scoped. Write agent brief.
- **needs-triage → ready-for-human**: Requires human judgement (design, product, security).
- **needs-triage → wontfix**: Rejected. Conservative — prefer needs-info over wontfix if uncertain.
- **ready-for-agent → ready-for-human**: Agent hit decision it cannot make. Delegate explicitly.
- **ready-for-agent → needs-triage**: Implementation revealed ambiguity. Re-triage.

### Agent Brief Template

Write when moving to **ready-for-agent**:

```markdown
## Agent Brief: #[issue] [Title]
Parent: #[parent-issue] | Label: ready-for-agent

### What to build
[Behavioral description, not procedural]

### Acceptance criteria
- [ ] Criterion 1
- [ ] Criterion 2

### Scope boundaries
- In scope: [list]
- Out of scope: [list]

### Reference
- Files: [paths — describe interfaces, not file paths]
```

### Out-of-Scope Knowledge Base

Enhancement marked **wontfix**:

1. Create `.out-of-scope/<concept>.md` with concept, decision, reason, prior requests → verify: file created with all fields
2. Link doc in closing comment → verify: comment posted
3. Subsequent requests for same feature → reference existing doc before closing → verify: doc referenced

## Tools

|------|----------|-------|--------|
| `gh` (issue tracker) | Gather, Apply | Issue ID, labels | Label applied, comment posted |
| `read` | Gather | Issue text, out-of-scope docs, code | Context |
| `grep` | Gather | Search patterns | Relevant code paths |
| `glob` | Gather | File patterns | Matching files |
| `edit` / `write` | Apply | Agent brief, out-of-scope doc | Updated/New documents |

## Examples

| Situation | Action | Outcome |
|-----------|--------|---------|
| Bug report with repro steps | Reproduce, confirm → ready-for-agent | Agent brief written |
| Feature request without spec | needs-info | Request specific details |
| Duplicate enhancement | Check out-of-scope → wontfix | Closing comment with reference |
| Maintainer override: "ship it" | Confirm → execute | State applied directly |

## Output

- Triage decisions with state transitions applied
- Agent briefs for issues promoted to ready-for-agent
- Out-of-scope decision records for wontfix enhancements
- Closing comments with explanations for every rejection

## Quality

- → verify: every AI comment starts with `> *This was generated by AI during triage.*`
- → verify: unlabeled → needs-triage first (never skip directly)
- → verify: needs-info returns to needs-triage when reporter replies
- → verify: maintainer override flags unusual transitions
- → verify: conservative with wontfix — prefer needs-info if uncertain
- → verify: never close without explanation (bug) or out-of-scope doc (enhancement)

## Protocol Shells

```
/protocol {
  intent="Route and prioritize incoming issue or request",
  input={ issue="<description>", context="<project-state>" },
  process=[ /decompose{type}, /compare{priority}, /synthesize{routing} ],
  output={ result="<triage-decision>", label="<routing-destination>" }
}
```

## Cognitive Tools

| Tool | When |
|------|------|
| /decompose | Break issue into bug/feature/question components |
| /compare | Evaluate priority against impact and urgency |
| /synthesize | Combine analysis into a routing recommendation |

## When NOT to Use

- Issues already in ready-for-agent with a valid agent brief
- Issues requiring immediate human judgement (design, product, security) — route directly to ready-for-human
- Issues confirmed outside project scope
