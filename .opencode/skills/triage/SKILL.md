## Use When

Incoming issues, bugs, or feature requests need triaging, prioritising, routing to right state.

## Core Concept

Every issue moves through states with clear transition rules:

```
unlabeled ──► needs-triage ──► needs-info ──► ready-for-agent ──► ready-for-human
                   │                                               │
                   └──────────► wontfix ◄──────────────────────────┘
```

| State | Description | Entry Action |
|-------|-------------|--------------|
| **unlabeled** | No state label applied | Apply `needs-triage` label |
| **needs-triage** | Needs evaluation | Attempt reproduction (bugs) or evaluate scope (enhancements) |
| **needs-info** | Waiting for reporter | Note what info needed; on reply → return to needs-triage |
| **ready-for-agent** | Fully specified, agent can implement | Write agent brief |
| **ready-for-human** | Needs human judgement | Note delegation blockers |
| **wontfix** | Rejected | Write out-of-scope doc or close with explanation |

## Precise Vocabulary

| Term | Definition |
|------|------------|
| **unlabeled** | Issue without state label |
| **needs-triage** | Needs evaluation — first actionable state |
| **needs-info** | Waiting for reporter response |
| **ready-for-agent** | Fully specified, agent can implement |
| **ready-for-human** | Requires human judgement |
| **wontfix** | Rejected |
| **agent brief** | Specification doc when moving to ready-for-agent |
| **out-of-scope doc** | Decision record when marking enhancement wontfix |

## Context Requirements

- Issue tracker with label management
- `.out-of-scope/` directory for rejection records
- Codebase access for reproduction and scope evaluation

## Workflow

### 1. Show What Needs Attention

3 buckets, oldest first:

- **Unlabeled** — new issues without state label
- **Needs-triage** — ordered by severity (bugs above enhancements)
- **Needs-info with activity** — reporter replied since last triage note

### 2. Triage a Specific Issue

1. **Gather context**: read full issue, search codebase, check `.out-of-scope/` for prior rejection
2. **Recommend category + target state**: classify `bug` or `enhancement`, propose state with reasoning
3. **Attempt reproduction (bugs only)**: reproduce → rule out environment → narrow to minimal repro
4. **Grill if uncertain**: ask clarifying questions one at a time until understood
5. **Apply outcome**: move to decided state

Every AI-generated comment MUST start with:

> *This was generated by AI during triage.*

### 3. Quick State Override

Maintainer says "move #42 to ready-for-agent" → trust, confirm, execute. Flag unusually permissive transitions (e.g. unlabeled → ready-for-agent) with brief note.

### State Transition Rules

- **unlabeled → needs-triage**: Required first step. Cannot skip.
- **needs-triage → needs-info**: More info needed. Be specific.
- **needs-info → needs-triage**: Reporter replied with requested info.
- **needs-triage → ready-for-agent**: Fully understood, specified, scoped. Write agent brief.
- **needs-triage → ready-for-human**: Needs human judgement (design, product, security).
- **needs-triage → wontfix**: Rejected. Prefer needs-info over wontfix if uncertain.
- **ready-for-agent → ready-for-human**: Agent can't decide. Delegate explicitly.
- **ready-for-agent → needs-triage**: Implementation revealed ambiguity. Re-triage.

### Agent Brief Template

Write when moving to **ready-for-agent**:

```markdown
## Agent Brief: #[issue] [Title]
Parent: #[parent-issue] | Label: ready-for-agent

### What to build
[Behavioral description, not procedural]

### Acceptance criteria
- [ ] Criterion 1
- [ ] Criterion 2

### Scope boundaries
- In scope: [list]
- Out of scope: [list]

### Reference
- Files: [paths — durable, describe interfaces not file paths]
```

### Out-of-Scope Knowledge Base

When enhancement marked **wontfix**, archive:

1. Create `.out-of-scope/<concept>.md` with:
   - Concept name
   - Decision: wontfix
   - Reason
   - Prior requests (issue numbers)

2. Link doc in closing comment

3. On subsequent same-feature requests, reference doc before closing

## Tool Requirements

- **terminal**: Issue tracker CLI (e.g. `gh`)
- **read**: Issue text, out-of-scope docs, codebase files
- **grep**: Search codebase for relevant paths
- **glob**: Find files
- **edit/write**: Create/update out-of-scope docs, agent briefs

## Output

- Triage decisions with state transitions
- Agent briefs for ready-for-agent issues
- Out-of-scope records for wontfix enhancements
- Closing comments with explanations for rejections

## Quality Criteria

- Every AI comment MUST start: `> *This was generated by AI during triage.*`
- unlabeled → needs-triage first (no skipping)
- needs-info returns to needs-triage on reply
- Maintainer can override (flag unusual transitions)
- Conservative with wontfix — prefer needs-info if uncertain
- Never close without explanation (bugs) or out-of-scope doc (enhancements)

## When NOT to Use

- Issues already in ready-for-agent with valid brief
- Issues needing immediate human judgement — route directly to ready-for-human
- Issues confirmed outside project scope
